# Самый маленький проект на Django

Мне не раз приходилось участвовать в обсуждениях "Django vs Flask" или сейчас более популярно "Django vs FastAPI".

Практически в каждом таком обсуждении звучит "Ну Django это комбайн, он огромный, даже самый маленький проект файлов десять".
Такое убеждение возникает из-за того, что "проект по умолчанию" на Django действителньо состоит из целого
набора разных файлов. Однако, все они создаются исключительно ради удобства и... не являются необходимыми!

А насколько небольшим на самом деле может быть проект на Django?

На самом деле минимальный размер проекта на Django - ничем не отличается от проекта на Flask или FastAPI. 
Это также ровно ОДИН файл - это settings.py. А что с остальными файлами?

- **manage.py** - этот файл нужен исключительно для удобства. Все что он делает - прописывает переменную окружения,
в которой указывается где собственно искать settings.py. но ее можно либо изначально там прописать, либо
передать через аргумент командной строки. Этот файл существует исключительно ради удобства - его можно удалить.
- **wsgi.py** - этот файл по сути делает то же самое, прежде вызвать get_wsgi_application. В целом, он нужен для того, чтобы
можно было легко обернуть wsgi приложение дополнительными middleware на уровне wsgi. Но это, в целом, нужно редко. 
Его тоже можно удалить.
- **asgi.py** - аналогично.
- **urls.py** - этот файл, как это ни странно, тоже не обязателен. В settings.py есть параметр ROOT_URLCONF, где он прописывается,
однако, есть и альтернативы. Можно указать не строку импорта, а:
  - Объект с атрибутом urlpatterns.
  - Непосредственно кортеж с url-шаблонами.
- **views.py** - не является частью проекта "по умолчанию", однако создается первым делом при написании первого же представления.
Однако это тоже не обязательно - представление можно описать прямо рядом с url-шаблоном.

Звучит интересно? А как же на практике?

Итак, вот пример минималистичного проекта на Django из одного файла - settings.py.

Это полноценный django-проект: с админой, базой данных, авторизацией, CSRF-защитой и т.д. И конечно "hello world" вьюхой!
Проверим его на практике?

## Поехали!
Так как проект из одно файла, зависимости нужно ставить самостоятельно: 
```commandline
pip install Django==6.0.7
```
Дальше обычные django операции. Запускаем миграции: 
```bash
python -m django migrate --settings=settings
```
Создаем суперпользователя:
```commandline
python -m django createsuperuser --settings=settings
```
Запускаем сервер:
```commandline
python -m django runserver --settings=settings
```
Открываем в браузере http://127.0.0.1:8000/admin/ и логинимся. Все работает!

## Deploy
Ок, а что с деплоем? В проде будет работать? Как насчет gunicorn без wsgi.py?
```commandline
docker build --tag 'smallest_django' .
docker run --rm -v ./data:/opt/data -p 127.0.0.1:8000:8000 smallest_django 
```
Снова открываем админку и она работает. Однако не подгружается статика: но так и должно быть.
При работе под gunicorn предполагается, что статику отдает nginx.
Однако для полноты демонстрации небольшой трюк:

```commandline
docker run --rm -v ./data:/opt/data -p 127.0.0.1:8000:8000 smallest_django gunicorn --bind 0.0.0.0:8000 "settings:get_application()"
```
Это заставит wsgi приложение отдавать статику также, как это делает runserver. Это просто чтобы показать,
что все работает и под gunicorn, и необязательность wsgi/asgi файлов.

## Еще меньше!

Мы уместились в один файл. Однако, если на него посмотреть - получилось все равно громоздко, особенно если
сравнивать с Flask/FastAPI. Это, конечно, от того - что функционал Django гораздо шире - он включает в себя
и админку, и продвинутый шаблонизатор, и разные middleware'ы, ORM. Собственно все то, что мы видим в этом
единственном файле.

Однако наша цель не найти оправдания размеру, а собрать минималистичный проект. Уравняем ситуацию, отказавшись 
от всего "лишнего"!

См. `smallest_settings.py`

```python
from django import urls
from django.http import HttpResponse

def home(_):
    return HttpResponse('Hello world from mini Django!')

ROOT_URLCONF = (
    urls.re_path(r'^$', home, name='home'),
)
```
Теперь по размеру он ничем не отличается от Flask или FastAPI. Такой же, совсем маленький файл.
Проверим его в работе?

```commandline
docker run --rm -v ./data:/opt/data -p 127.0.0.1:8000:8000 -e DJANGO_SETTINGS_MODULE="smallest_settings" smallest_django
python -m django runserver --settings=smallest_settings
```

## Выводы

Да, конечно если сравнивать с Flask или FastAPI - Django это конечно огромный комбайн, который включает в 
себя много всего. Однако это хорошо спроектированный модульный фреймворк, где практически все части - отделяемые.

"Базовая структура" проекта на Django сильно больше чем на F/F, однако это сделано исключительно из удобства
разработки, из предположения, что ваш проект будет больше чем на один-два файла. Потому что большинство проектов
такие. И даже если ваш проект сейчас не такой - скорее всего он таким станет.

Однако, если вы хотите сделать маленький проект из одного модуля - на Djangо это тоже не проблема. 
И в этом плане между основными фреймворками, по сути, нет никакой разницы.
